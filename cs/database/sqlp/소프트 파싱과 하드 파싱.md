# sql-soft-parsing-vs-hard-parsing

## 개념

SQL은 실행 전에 파싱 과정을 거친다.  
이때 SQL 문법과 객체 유효성을 확인하고, 실행 계획을 만들거나 재사용한다.

소프트 파싱과 하드 파싱의 차이는 `기존 실행 계획을 재사용하는가`에 있다.
소프트파싱은 예전 실행 계획을 재사용, 하드파싱은 새로 만든다.

| 구분 | 소프트 파싱 | 하드 파싱 |
| --- | --- | --- |
| 의미 | 기존에 생성된 실행 계획을 재사용 | 새로운 실행 계획을 다시 생성 |
| 비용 | 상대적으로 작음 | CPU와 메모리 비용이 큼 |
| 라이브러리 캐시 접근 | 재사용 중심 | 탐색 후 새로 적재 |
| 성능 영향 | 일반적으로 유리 | 반복되면 성능 저하 가능 |

### 라이브러리 캐시

-  SGA(System Global Area)의 구성요소, SGA는 서버 프로세스와 백그라운드 프로세스가 공통으로 엑세스하는 데이터와 제어 구조 캐싱하는 메모리 공간
- 라이브러리 캐시: SQL 파싱, 최적화, 로우 소스 생성 과정을 거쳐서 생성한 내부 프로시저를 반복 재사용할 수 있도록 캐싱해 두는 메모리 공간

### 소프트 파싱 , 하드 파싱

전체 구조
- 사용자가 SQL문 전달
- DBMS가 SQL 파싱
- 해당 SQL이 라이브러리 캐시에 존재하는지 확인
- 캐시에서 찾으면 바로 실행, 못하면 최적화 -> 로우 소스 생성 -> 실행

- 소프트 파싱: 캐시에서 찾는데 성공해서 바로 실행단계로
- 하드 파싱: 캐시에서 찾는데 실패해서 최적화, 로우소스 생성 과정을 거침




## 라이브러리 캐시가 필요한 이유

데이터베이스는 같은 SQL이 반복 실행되는 환경에서 매번 실행 계획을 새로 만들면 비효율적이다.  
그래서 가능한 경우 기존 실행 계획을 재사용해서 CPU 사용량과 파싱 시간을 줄인다.

실무에서는 조회 자체보다 파싱이 병목이 되는 경우도 있다.  
특히 짧고 자주 호출되는 SQL이 많은 서비스에서는, 라이브러리 캐시를 거치는 소프트 파싱 유도가 중요하다

## 문제 상황

다음과 같은 상황에서는 하드 파싱이 자주 발생할 수 있다.

- SQL 문자열이 매번 다르게 생성되는 경우
- 바인드 변수를 쓰지 않고 리터럴 값을 직접 붙이는 경우
- 공유 풀에서 기존 실행 계획을 재사용하지 못하는 경우
- 유사한 SQL이 대량으로 생성되는 경우

예시:

```sql
SELECT * FROM member WHERE id = 101;
SELECT * FROM member WHERE id = 102;
SELECT * FROM member WHERE id = 103;
```

위처럼 값만 바뀌는데도 SQL 문자열 자체가 달라지면 하드 파싱 가능성이 커진다.

SQL 옵티마이저는 순식간에 많은 연산을 한다. 그 중 I/O 작업에 제일 집중되지만, 
하드파싱은 CPU를 많이 소비한다. 
이렇게 CPU를 많이 소비하는 작업을 거쳐서 생성한 내부 프로시저를 한 번만 사용하고 버린다면 비효율적이기 때문에 라이브러리 캐시를 사용해야 한다. 


## 해결 방식

핵심은 `같은 형태의 SQL을 반복 사용`하도록 만드는 것이다.

### 1. 바인드 변수 사용

```sql
SELECT * FROM member WHERE id = :id;
```

바인드 변수를 사용하면 SQL 텍스트를 재사용하기 쉬워지고, 실행 계획 공유 가능성이 높아진다.


#### 1.1 이름없는 SQL 문제

- SQL은 이름이 없기 때문에 전체 텍스트가 이름 역할을 함
- 처음 실행 시 생성된 내부 프로시저를 라이브러리 캐시에 적재해서 재사용
- 캐시 공간이 부족해지면 버려졌다가 다시 실행 시 캐시에 적재됨
- 이름이 없으므로 텍스트 중 작은 부분이라도 수정되면 다른 객체가 새로 생성되는 구조
- 오라클의 경우 sql id가 있지만 sql 텍스트가 변하면 id도 변함

#### 1.2 공유 가능 SQL 문제

- 같은 의미를 가지는 SQL문이더라도, 실행 시 별도의 캐시 공간을 사용함
- 한번에 일괄로 100만개의 데이터를 조회하는 SQL문을 실행한다고 할 때, 의미가 같음에도 불구하고 SELECT 문 하나를 실행할 때마다 DBMS 내부 프로시저를 하나씩 만들어서 라이브러리 캐시에 적재함
- 이렇게 모든 SQL을 하드파싱으로 실행하게 되면, I/O작업이 거의 없음에도 불구하고 CPU 사용량이 급증하게 됨

```SQL
SELECT * FROM MEMBER WHERE MEMBER_ID = 1;
SELECT * FROM MEMBER WHERE MEMBER_ID = 2;
SELECT * FROM MEMBER WHERE MEMBER_ID = 3;
...
SELECT * FROM MEMBER WHERE MEMBER_ID = 1000000;

```

-> 해결 방법: 바인드 변수를 사용
- 파라미터 Driven 방식으로, ID만을 파라미터로 받는 프로시저 하나를 100만개의 SELECT문 간에 공유하도록 함
	```java
	String sql = "select * from member where member_id = ?";
	PreparedStatement pstmt = con.prepareStatement(sql);
	pstmt.setLong(1, memberId);
	ResultSet rs = pstmt.executeQuery();

	```
- 라이브러리 캐시를 조회해 보면, 하나의 SQL만 발견되게 됨 
```sql
SELECT * FROM MEMBER WHERE MEMBER_ID = :id;
```


### 2. 애플리케이션에서 동적 SQL 남용 줄이기

- 조건마다 문자열을 직접 이어 붙이는 방식을 줄인다.
- PreparedStatement, ORM 파라미터 바인딩을 적극 활용한다.

```java
// sql 텍스트 고정
String sql = "select * from orders where status = ? and member_id = ?";

// 실행 시 값만 변경
PreparedStatement pstmt = con.prepareStatement(sql);
pstmt.setString(1, status);
pstmt.setLong(2, memberId);

ResultSet rs = pstmt.executeQuery();

```

#### ORM 파라미터 바인딩 예시 - JPQL

```java
@Query("select o from Order o where o.user.id = :userId")
List<Order> findByUserId(@Param("userId") Long userId);
```
JPQL

#### 예외 - QueryDSL

QueryDSL은 동적 쿼리인데도, 이어 붙이더라도 바인딩 방식이라는 장점이 있다. 
다만 조건 조합 수가 너무 많아지면 실행 계획 캐시 관점에서 SQL 종류가 많아지기 때문에, 조건을 항상 분명하게 설정해야 한다. 
검색용 쿼리를 만들 때는 조건이 많아지면 역할별로 분리하는 방법으로 쪼개는 것이 권장된다. 

### 3. 공유 풀과 실행 계획 재사용 관점으로 점검

- DB가 같은 SQL로 인식하는지 확인한다.
- 라이브러리 캐시 miss가 과도한지 본다.

## 장단점

| 항목 | 장점 | 단점 |
| --- | --- | --- |
| 소프트 파싱 | 파싱 비용 감소, CPU 절감, 응답 시간 안정화 | 계획 재사용이 항상 최적 계획을 보장하는 것은 아님 |
| 하드 파싱 | 현재 조건에 맞는 새 계획 생성 가능 | 반복되면 CPU 낭비, 동시성 저하, 성능 흔들림 |

## 트레이드오프

소프트 파싱은 일반적으로 성능상 유리하지만, 모든 상황에서 무조건 최선은 아니다.

| 관점 | 소프트 파싱 중심 | 하드 파싱 발생 가능 |
| --- | --- | --- |
| 성능 | 반복 호출 환경에서 유리 | 호출 빈도가 높으면 불리 |
| 유연성 | 기존 계획 재사용 | 상황에 따라 새 계획 수립 가능 |
| 운영 안정성 | 예측 가능한 응답 시간에 유리 | 파싱 부하가 커지면 시스템 전체에 영향 |

실무에서는 보통 `하드 파싱을 줄이는 방향`으로 설계하되, 바인드 변수 사용에 따른 실행 계획 이슈도 같이 점검한다.

## 실무에서의 사용

- 트래픽이 높은 API에서는 같은 형태의 SQL을 반복 실행하는 경우가 많다.
- 이때 하드 파싱이 많으면 DB CPU 사용량이 먼저 튀는 경우가 있다.
- ORM을 사용하더라도 동적 조건이 많으면 SQL이 지나치게 다양해질 수 있다.

실무에서는 다음을 같이 본다.

- PreparedStatement 사용 여부
- 바인드 변수 사용 여부
- SQL 종류 수가 비정상적으로 많은지
- 라이브러리 캐시 miss 또는 parse call 비율

## 주의할 점

- 소프트 파싱이 된다고 해서 무조건 빠른 SQL이라는 뜻은 아니다.
- 느린 SQL은 실행 계획 자체가 비효율적일 수 있다.
- 하드 파싱 문제와 인덱스/조인 문제를 분리해서 봐야 한다.

## DB 쿼리에서의 적용

- 조건 조합을 너무 많게 하지 않기
- 비효율적인 조건을 넣지 않기 
- 쿼리 하나에 책임이 너무 많아지도록 하지 않기
- 인덱스 활용 가능성도 같이 보기, 인덱스가 깨지지 않도록 조심하기

### QueryDSL에서 좋은 쿼리문 작성하기

1. 유스케이스 분리
2. 공통 조건과 유스케이스 조건 분리
	- BooleanExpression을 사용해서 공통 조건, 검색용 조건을 분리하기
3. 사용자별로 분리(관리자,사용자)
	- 합치지 말고 사용자별로 분리된 쿼리문 작성하기
4. 검색 조건 객체도 역할별로 분리
	- 검색용 DTO에서 필요한 필드만 넣도록 하기
5. 복잡한 조건은 조건 묶음으로 분리 - BooleanBuilder 사용
